遅延キューを使って処理を遅らせる
遅延キューは、メッセージやタスクを即座に実行するのではなく、指定した時間が経過してから処理を開始する仕組みです。通常のキューがFIFO（First In, First Out）の原則で動作するのに対し、遅延キューは時間という要素を加えて処理のタイミングを制御します。システムデザイン面接において、遅延キューを頭にいれておくことで。リアルタイムで処理する必要がないタスクや、意図的に処理を遅らせたいタスクを効率的に管理できます。
遅延キューとは
遅延キューは、メッセージに実行開始時刻を設定し、その時刻になるまで処理を保留する仕組みです。通常のキューでは、メッセージがキューに追加されると即座にワーカーが処理を開始しますが、遅延キューでは指定された遅延時間が経過するまでメッセージは取得できない状態になります。
例えば、AmazonSQSの遅延キューでは、メッセージ送信時に遅延秒数を指定することでコンシューマーへの配信を最大は15分間遅延できます。
Rabbit MQでもプラグイン（RabbitMQ Delayed Message Plugin）を活用することで、メッセージにx-delayのカスタムヘッダーを追加するとそのmilliseconds分配信が遅延されます。
また、LINEではRedis Sorted Setsを使った場合は、実行開始時刻をスコアとして設定し、現在時刻以下のスコアを持つメッセージを取得することで遅延実行を実装できます。

システムデザイン面接で遅延キューを使う時
遅延キューを使用するかは必要な遅延時間の長さによって大きく変わります。面接では、要件に応じて最適な手法を選択できることを示しましょう。
数分程度の短い遅延では、メッセージキューの遅延キューを使うことを考えましょう。例えば、ユーザーの操作に対する確認処理や、API呼び出しの再試行処理など、比較的短時間で完了する処理に適しています。面接官から「注文後5分以内にキャンセルできる機能」について質問された場合、この手法を提案できます。
数時間から数日といった長い遅延では、遅延キューというよりはスケジューラと考えるとよさそうです（広義的にはどちらも同じ意味）。この場合、実行予定時刻をデータベースに保存し、定期的にポーリングして実行すべきタスクをチェックし実行します。
面接では、この判断基準を明確に説明することで、システム設計における実践的な知識があることを示せます。遅延時間が長くなるほど、永続化の重要性が高まることや、メモリ使用量とのトレードオフを理解していることをアピールしましょう。

ページネーションを設計する
大規模データセットに対してクライアントが部分的にデータを取得する際、ページネーションは非常に重要な設計技術です。無制限に全件を返してしまうと、パフォーマンスやネットワークに悪影響を与えるだけでなく、可用性やスケーラビリティにも悪影響を及ぼします。
本章では代表的な2つのページネーション方式、オフセット形式とカーソル形式について説明し、それぞれの利点と欠点を分析します。
オフセット形式（Offset Pagination）
オフセット形式はもっとも単純なページネーション方式です。典型的には以下のようなクエリで実現されます。
リクエスト例
GET /api/items?limit=20&offset=40
{
"total": 5230,
"limit": 20,
"offset": 40,
"items": [
{ "id": 41, "name": "Item 41", "created_at": "2025-06-01T10:00:00Z" },
{ "id": 42, "name": "Item 42", "created_at": "2025-06-01T09:59:59Z" },
...
]
}
SQL例
SELECT _ FROM items ORDER BY created_at DESC LIMIT 20 OFFSET 40;
この例では、itemsテーブルから41件目から60件目までのデータを取得しています。
利点
実装が簡単です。多くのRDBが標準でサポートしており、インデックスが適切に張られていればそこそこのパフォーマンスが得られます。
ユーザーに「ページ数」や「総ページ数」をわかりやすく表示できます。
欠点
大量データに弱い: OFFSETが大きくなるとスキャンコストが高くなります。たとえば100,000件目を表示しようとすると、RDBは99,999件スキップする必要があります。次の記事の計測結果によるとMySQLにおいて、OFFSETが大きくなるにつれて線形でクエリ時間が長くなっている事がわかります。

不安定な順序: データが同時に挿入・削除されるような環境では、同じクエリでも異なるデータが返ることがあります。ユーザーが「前のページに戻る」と意図した動作にならないことがあります。
カーソル形式（Cursor Pagination）
カーソル形式では、前回取得したデータの一意な値を使って次のデータを取得します。典型的には以下のような形式になります。
リクエスト例
GET /api/items?limit=20&cursor=01BX5ZZKBKACTAV9WEVGEMMVRY
この例では、idがカーソルとして使われています。IDは降順で並んでおり、このカーソルよりも小さいIDのデータを取得するという意味になります。
{
"total": 5230,
"limit": 20,
"items": [
{ "id": "~~~", "name": "Item 41", "created_at": "2025-06-01T10:00:00Z" },
{ "id": "~~~", "name": "Item 42", "created_at": "2025-06-01T09:59:59Z" },
...
],
"next_cursor": "01BX5ZZKBKACTAV9WEVGEMMVRT"
}
next_cursorとして返されたIDを使って次ページをリクエストしていきます。
SQL例
SELECT _ FROM items WHERE id < '01BX5ZZKBKACTAV9WEVGEMMVRY' ORDER BY id DESC LIMIT 20;
シーケンシャルなIDというと AUTO_INCREMENT を思い浮かべる方もいるかもしれませんが、シャーディングや分散システムでは欠点も多く、代わりにULID（Universally Unique Lexicographically Sortable Identifier）がよい選択となります。ULID は先頭48ビットをミリ秒単位のタイムスタンプ、後ろ80ビットにランダム成分を持つため、分散システムにおいても中央制御なしでK-sorted sequence（おおまかな順序）な時系列IDの生成が可能です。

利点
スケーラブル: インデックスが張られていれば、カーソル条件に合致するデータの絞り込みは高速に行えます。
安定した順序: カーソルが状態を明示的に持っているため、途中でデータが追加・削除されても意図しないページのずれが起こりにくいです。
欠点
ページ番号の概念がない: ユーザーに「10ページ目を見たい」と言われても対応が難しくなります。
トレードオフ比較
オフセット形式とカーソル形式には明確なトレードオフがあります。
少量データ or 明確なページ番号が必要な場合はオフセット形式が向いています。
大量データ or 順序が重要な場合、あるいは更新が頻繁に行われるシステムではカーソル形式のほうが堅牢です。
たとえば、
商品一覧や検索結果のようにページ番号が重要な場合 → オフセット形式
SNSフィードやチャットメッセージのように順序やリアルタイム性が重視される場合 → カーソル形式

全文検索を導入する
大規模システムにおける全文検索機能の設計について体系的に解説します。システムデザイン面接で「なぜ全文検索が必要なのか」「どのように効率的な検索を実現するのか」「分散環境での検索システムをどう構築するか」を論理的に説明できるようになることを目指します。しかし、多くのポジションは自然言語処理や全文検索の専門家を集めているわけではないため、基本的は要素だけ抑えておくことで実際の面接では十分です。
どのような時に活用するのか？
全文検索は主に次のような場面で必要となります。
大量のテキストデータから特定の情報を素早く見つける必要がある: ECサイトの商品検索、記事検索、チャットメッセージ検索など
部分一致や関連性の高い結果を返したい: 完全一致検索では見つからない関連コンテンツも含めて検索結果を提供する
システムデザインでは、まず単純なLIKE検索で十分ではない場面が多いです。この場合は全文検索システムの導入を検討するとよいでしょう。
インデックスを使わない場合の問題
全文検索においてインデックスを使わない場合、検索のスケーラビリティで問題が発生します。検索クエリに対してすべてのドキュメントを順次スキャンする場合を考えましょう。この線形検索の計算量はO(n×m)となります。ここでnはドキュメント数、mは各ドキュメントの平均文字数です。
例えば、100万件のドキュメントがあり、各ドキュメントが平均1000文字の場合、1つの検索クエリで最大10億文字の走査が必要になります。これは現実的な応答時間を大幅に超えてしまいます。
転置インデックスとは
転置インデックスは全文検索システムの核となるデータ構造です。各単語がどのドキュメントに出現するかを記録します。このデータ構造は単語をキーとし、その単語が出現するドキュメントのリストを値とするマップ構造です。各エントリには以下の情報が含まれます。
ディクショナリ: インデックスのキーとなる検索対象の語句
ポスティングリスト: その単語が出現するドキュメントのID一覧
例えば、「システム設計の基本」というdoc1と「設計パターンの応用」というdoc2がある場合、転置インデックスは次のようになります。これにより、ユーザーがクエリとして含めた単語（たとえば「設計 パターン」など）のポスティングリストを取り出して集合演算（AND／OR）するだけで、関連ドキュメントを瞬時に絞り込めます。
なお、「の」がインデックスにないのは後に解説するStop Wordsを除去しているからです。

インデックス作成の前処理技術
生のテキストデータをそのままインデックス化するのではなく、適切な前処理を行うことで効率的な検索を実現できます。ここではシステムデザインの面接に十分な主要な前処理技術について解説します。
Tokenization（トークナイゼーション）
Tokenization（トークナイゼーション）は、テキストを「検索可能な単語（トークン）」に分割する処理です。インデックス作成の最初のステップであり、他のすべての前処理（ストップワード除去、ステミング、N-gram 生成など）の前提となります。
例えば、以下の文：
"Scalable systems are essential for global applications."
は、Tokenizationによって次のように単語に分解されます：
["Scalable", "systems", "are", "essential", "for", "global", "applications"]
言語や用途によってトークン化の方法は異なり、特に英語のような空白区切り言語では比較的単純ですが、句読点やハイフンなどへの対応も重要です。
Tokenizationの段階で行われる代表的な処理には以下が含まれます：
小文字化（Lowercasing）：大文字・小文字を区別しないよう "Systems" → "systems" に統一
記号の除去や正規化：不要な句読点を削除または分離（e.g., "scalable," → "scalable"）
略語や略記の展開：必要に応じて "can't" → "can", "not" のように分割
例えば、ElasticsearchではStandard tokenizerがこの処理を行い、正規表現や記号の扱いも含めて高精度なTokenizationを実現します。

Stop Words（ストップワード）除去
Stop Wordsは検索において意味を持たない、または検索価値が低い単語のことです。日本語では「は」「が」「を」「に」などの助詞、英語では「the」「and」「or」「but」などの冠詞や接続詞が該当します。
例えば、”the design of a global system and its scalability requirements”というフレーズからStop Words（the, of, a, and, its など）を取り除くと、”design global system scalability requirements”だけがインデックスに残ります。
基本的にどの検索ユースケースにおいてもスStop Wordsの除去は有効です。
これにより、
インデックスサイズの削減: Stop Wordsは文書中に高頻度で出現するため、これらを除去することでインデックスサイズを大幅に削減できます。
検索精度の向上: Stop Wordsを含めた検索では、関連性の低いドキュメントが上位に表示される可能性があります。除去することで、より意味のある検索結果を提供できます。
処理速度の向上: インデックスサイズが小さくなることで、検索処理の速度が向上します。
例えば、ElasticsearchではStop token filterがこの役割を担います。

Stop token filter | Reference
Stemming（ステミング）
Stemmingは単語を語幹部分まで削り、異なる活用形や派生形を同じ語として扱う技術です。英語では「running」「runs」「ran」をすべて「run」に変換します。
これにより、
検索漏れの防止: ユーザーが「run」で検索した場合でも、「running」を含むドキュメントを見つけることができます。
インデックス効率の向上: 複数の活用形を1つの語幹で表現することで、インデックスサイズを削減できます。
例えば、ElasticsearchではStemmer token filterがこの役割を担います。

Stemmer token filter | Reference
Lemmatization（レマタイゼーション）
Lemmatizationは単語を辞書の見出し語形に変換する技術です。Stemmingよりも言語学的に正確で、文脈を考慮した変換を行います。
StemmingとLemmatizationの違い
精度の違い: Stemmingは機械的な規則適用のため、「better」を「better」のまま残してしまいますが、Lemmatizationは「good」に変換できます。
処理速度の違い: Stemmingは高速ですが、Lemmatizationは辞書参照や文脈分析が必要なため処理時間がかかります。
N-gram
N-gramとは、テキストを「連続するN個の文字または単語」ごとに分割してインデックス化する手法です。次の2種類がよく使われます。なお、N=1はunigram、N=2はbigram、N=3はtrigramと呼ばれます。

ユーザーを認証・認可する
システムデザイン面接において、ユーザー認証と認可は無視して良い場面があります。しかしこれらの概念は、Web APIの設計においても重要な役割であり、少なくともここに書いている程度の認証認可の仕組みに関しては抑えられるようにしておきましょう。
ユーザー認証・認可の基本概念
認証とは、システムやサービスにアクセスする際に、ユーザーが正当な権限を持つ者であることを確認するプロセスです。これにより、不正アクセスを防止し、データやリソースのセキュリティを確保します。
認証と密接に関連するのが認可であり、これは認証されたユーザーが特定のリソースや操作にアクセスする権限を持つかどうかを判断するプロセスです。Web APIにおいては、これらのプロセスが特に重要であり、適切な認証・認可メカニズムを設計することが求められます。
Cookieを利用した認証・認可
Cookieは、クライアント側に保存される小さなデータで、ユーザーのセッション情報を保持するために使用されます。Webアプリケーションでは、ユーザーがログインすると、サーバーは認証情報を含むCookieをクライアントに送信します。
📗
リクエストを送信（HTTP）
でも解説しているので合わせて読んでおくと良いです。
以降のリクエストでは、このCookieが自動的に送信され、サーバーはその情報を基にユーザーを認証・認可します。Cookieを利用した認証・認可の利点として、セッション管理の容易さや、ブラウザが自動的にCookieを送信する特性を挙げることができます。
具体例を上げると例えばサーバーサイドで認証のトークンxxxxxxを生成した場合、サーバーからのレスポンスヘッダーにSet-Cookie を含めます。
Set-Cookie: token=xxxxxx; HttpOnly; Secure; SameSite=Strict; Path=/
HttpOnly属性により、JavaScriptからのアクセスが制限され、Secure属性により、HTTPS接続時のみ送信されます。SameSite=Strict属性は、クロスサイトリクエスト時のCookie送信を制限し、CSRF攻撃のリスクを低減します。
Set-Cookieヘッダーを含んだレスポンスを受け取ったブラウザが自動的にクッキーを保存し、以降は有効期限が切れたり削除されるまでCookieヘッダーに自動的に送信するようになります。

Authorizationヘッダーを利用した認証・認可

JWT（JSON Web Token）について
JWTとは、JSON Web Tokenの略で、情報を安全に送受信するためのオープンスタンダード（RFC 7519）です。
JWTは数多くの企業で実際に使われており近年では採用していない企業の方が珍しいでしょう。例えばLINEヤフー（旧ヤフー）では、ユーザーのクレデンシャル情報や、Yahoo! ID連携 v2のID TokenなどにJWTが活用されています。

JWTは、ヘッダー、ペイロード、署名の3つの部分で構成され、これらを組み合わせてトークンを生成します。JWTは署名付きのため改ざんできない、ペイロードの中にユーザーの認証認可の情報、有効期限などの情報を入れ込むことができる事ができます。これにより、従来ではセッションテーブルなどを用意してトークンをサーバー側で永続化する必要がありましたが、その必要はありません。
好きな場所で認証認可が可能なため、例えばマイクロサービスの各サービスも可能ですし、日本経済新聞社ではFastly（CDN）レイヤーでJWTをデコードして、認証認可の実装を行っています。

SaaSのマルチテナントを設計する
SaaS企業では複数の顧客を効率的にサービス提供するため、マルチテナント設計が重要な技術要件となります。SaaSにおけるテナントとは、システムが扱う顧客の論理的な単位を指します。これは契約の単位や課金が発生する単位、データのスコープを決める重要な概念です。一般的には企業や事業部といった単位がテナントに該当し、各テナント内に複数のユーザーが所属する構造になります。
例えば、プロジェクト管理のSaaSでは会社単位で契約し、社員一人ひとりがメンバーとして利用する形態が考えられます。この場合、会社がテナント、社員がユーザーという関係になります。特に日本のSaaS企業を受けてシステムデザイン面接にあたった際には出題される傾向が高いです。
SaaSのパーティションニングモデル
パーティショニングモデルは、SaaSにおいてテナントのデータや環境をどのように分離・管理するか定義します。主要なモデルとして、サイロモデル、ブリッジモデル、プールモデルの3つがあります。これらのモデルは、それぞれ異なる分離レベルとコスト効率、運用複雑性を持っています。一般的に分離度としてはサイロモデル > ブリッジモデル > プールモデルの順で高いです。
システム設計の要件ごとに最適なモデルは異なるため、各モデルのメリット・デメリットを理解して選択することが重要です。面接官と話し合いながら決めていきましょう。
サイロモデル（シングルテナント）
サイロモデルは、テナントごとに専用の環境を用意して提供するアプローチです。1つのデータベース上に1つのテナントデータを作成し、同一構成のデータベースが複数存在する形になります。OSリソースであるCPU、メモリ、ディスクが他のテナントから完全に分離されています。
このモデルの最大のメリットは、物理的に分離されているためテナント毎に独立した管理が可能になることです。他テナントのデータ操作リスクが限りなく低く、テナント固有でスケールアップや可用性設計を柔軟に行えます。また、データベース障害時の影響を特定のテナントのみに限定できるため、システム全体の堅牢性が向上します。
一方でデメリットとして、管理対象が増えるためコストが高くなります。例えば、1000テナント存在する場合、データベースやアプリケーションが完全分離されているため、データベースマイグレーションやデプロイを行うだけで大きなコストを支払うことになります。

ブリッジモデル（スキーマ分離 / テーブル分離）
ブリッジモデルは、1つのデータベース上にテナント毎のスキーマやテーブルを作成するアプローチです。ちなみにスキーマとはPostgreSQLでは名前空間のようなもので同じテーブル名であっても名前空間が異なれば違いものとして扱われます。同一構成のスキーマ/テーブルが複数存在し、CPU、メモリ、ディスクは他のテナントと共有します。サイロモデルとプールモデルの中間的な位置づけのモデルです。
このモデルのメリットは、サイロモデルに比べ運用効率化やコスト削減が可能になることです。メンテナンス（テーブル構成変更など）が容易で、各テナントの余剰リソースを共有できます。また、テナント横断の監視や管理がシンプルになり、運用負荷が軽減されます。
一方でデメリットとして、依然としてテナントごとにテーブル数が増加していきます。これによりテーブルのマイグレーションのコストも高いです。また、環境が分離されていないため、ノイジーネイバー問題やテナント個別の監視・チューニングの困難さが新たに登場します。ノイジーネイバー問題とはある利用量の大きなテナントが、他の利用量の少ないテナントに影響を及ぼす事象を指します。例えば、テナントAが高負荷のクエリを発行している場合に、テナントBは何も高負荷な利用を行っていないにも関わらずパフォーマンスの問題に直面します。
ノイジーネイバー問題に対しては、各テナントごとにレート制限を導入する、一部のビックテナントをサイロモデルにする、データベースのパーティショニングで上手く分離するなどが考えられるアプローチです。

プールモデル
プールモデルは、全てのテナントデータを1つのスキーマやテーブルで共有するアプローチです。テナントデータへのアクセスを制御するためのIDで管理し、CPU、メモリ、ディスクは他のテナントと共有します。最も共有度の高いモデルです。
メリットとして、メンテナンス作業が最も容易になります。ブリッジモデルより更にデータの冗長性が低く、各テナントの余剰リソースを最大限共有できます。テナント横断の監視や管理が最もシンプルになり、運用効率が向上します。
最大のデメリットは、クロステナントアクセスのリスクが高いことです。アプリケーションの実装ミスにより、他テナントのデータにアクセスしてしまう可能性があります。これは重大なセキュリティインシデントにつながるため、十分な対策が必要です。また、ブリッジモデルと同様に環境を共有しているので、ノイジーネイバー問題やテナント個別の監視・チューニングの困難さも課題です。
例えば、テーブルにテナントIDカラムを追加し、WHERE句でのフィルタリングやRLS（行レベルセキュリティ）機能を使用してデータアクセスを制御します。RLS（行レベルセキュリティ）に関しては次のセクションで詳しく解説します。実際にSansanではRLSを利用したプールモデルに移行しています。

RLS（行レベルセキュリティ）
RLS（行レベルセキュリティ）とは、データベースのテーブル内で、ユーザーやロールに基づいて特定の行のみにアクセスを制限するセキュリティ機能です。SQL Server、PostgreSQLなどで活用できます。
具体的な例を見たほうが分かりすいので、会社Aと会社Bのテナントにそれぞれ顧客がいたとしましょう。RLSを利用すると、会社Aのユーザーは会社Bのレコードを取得できない、逆に会社Bのユーザーは会社Aのレコードを取得できないようにできます。

1. テーブル作成とRLS有効化（PostgreSQLの例）
   -- 顧客テーブル作成（company_idでテナント分離）
   CREATE TABLE customers (
   id SERIAL PRIMARY KEY,
   name VARCHAR(100),
   email VARCHAR(100),
   company_id INTEGER -- テナントID
   );

-- サンプルデータ
INSERT INTO customers (name, email, company_id) VALUES
('田中太郎', 'tanaka@companyA.com', 1), -- 会社A
('佐藤花子', 'sato@companyA.com', 1), -- 会社A
('山田次郎', 'yamada@companyB.com', 2), -- 会社B
('鈴木美咲', 'suzuki@companyB.com', 2); -- 会社B

-- RLS有効化
ALTER TABLE customers ENABLE ROW LEVEL SECURITY; 2. ポリシー作成
-- テナント分離ポリシー（自社のデータのみアクセス可能）
CREATE POLICY tenant_isolation_policy ON customers
FOR ALL
USING (company_id = current_setting('app.current_company_id')::INTEGER); 3. 使用例
-- 会社Aでログイン
SET app.current_company_id = 1;
SELECT \* FROM customers;
-- 結果: 田中太郎、佐藤花子のみ表示

-- 会社Bでログイン
SET app.current_company_id = 2;
SELECT \* FROM customers;
-- 結果: 山田次郎、鈴木美咲のみ表示

-- データ挿入も自動的に制限される
INSERT INTO customers (name, email, company_id) VALUES ('新規顧客', 'new@test.com', 1);
-- 会社Bでログイン中なら、company_id=1のデータは見えない
