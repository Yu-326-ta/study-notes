大量のイベントデータをバッチ処理する
システムデザインで大量のイベントデータをバッチ処理するユースケースとしては、リアルタイム性が要求されないデータを集計することが多いです。イベントデータとしてはユーザーの行動ログ、システムメトリクス、IoTデバイスからのセンサーデータなどがあげられます。この章では、大量のデータを分散バッチ処理する技術や、どのように扱うかについて離していきます。
MapReduceとApache Hadoop
MapReduceは、2004年にGoogleから発表されたデータ処理の代表的なモデルです。彼らの論文「

MapReduce: Simplified Data Processing on Large Clusters
」において、関数型プログラミングでよく使用されるmap関数とreduce関数から着目を得て開発されたものです。
MapReduceを使ったフレームワークとしてはApache Hadoopがあり、Hadoop Distributed File System（HDFS）に格納されたビッグデータにアクセスするために使用されます。
MapReduceの登場によって、大規模なデータセットを複数のマシンに分散して効率的に処理することが可能となり、従来は高価な専用ハードウェアに依存していたビッグデータ処理が、安価な汎用サーバーのクラスタ上でも現実的になりました。後に登場する様々な分散データ処理フレームワークに影響を与えた重要な技術です。

1. Mapフェーズ
   Mapフェーズでは、入力データを小さな単位に分割し、各データ片に対して同じ処理を並列実行します。この段階では、各ノードが独立してデータを変換し、中間結果を生成します。
   例えば、ECサイトの売上データからカテゴリ別の売上を集計する場合を考えてみましょう。Mapフェーズでは、各売上レコードから「カテゴリ（electronics、clothing、books）」と「売上金額」のペアを抽出します。このとき、売上データが複数のノードに分散されていても、各ノードが独立して同じ処理を実行できます。
2. Shuffleフェーズ
   Shuffleフェーズでは、Mapフェーズで生成された中間結果を、キーに基づいて適切なノードに再配布します。同じキーを持つデータが同一ノードに集約されるよう、ネットワーク経由でデータの移動が行われます。
   先ほどの例では、同じカテゴリに関するデータがすべて同一ノードに集められます。例えば、「electronics」に関するすべてのアクセス記録が特定のノードに送られます。
3. Reduceフェーズ
   Reduceフェーズでは、同じキーを持つ値のリストに対して集約処理を実行します。この段階で最終的な結果が生成されます。
   先ほどの例では、各ノードが受け取った同一カテゴリの売上を合計して、最終的なカテゴリ別の売上合計を計算します。

  MapReduceの問題とApache Spark
  MapReduceは大量データの分散処理において重要な役割を果たしてきましたが、実際の運用では複数の制約と課題が明らかになりました。これらの問題を解決するために、より柔軟で高性能なデータ処理エンジンが開発されています。
  MapReduceの主な問題
  中間結果のディスク書き込みによる性能劣化
  MapReduceの最大の問題は、Map処理とReduce処理の間で中間結果を必ずディスクに書き込む必要があることです。Shuffleフェーズでデータを再配布する際に、全ての中間データがファイルシステムに保存されるため、ディスクI/Oがボトルネックとなります。
  固定的な処理モデルの制約
  MapReduceは「Map → Shuffle → Reduce」という固定的な2段階の処理モデルに制限されています。複雑なデータ分析では、複数のMapReduce処理を連鎖的に実行する必要があり、各段階でディスクI/Oが発生します。
  例えば、顧客の購買データから「商品カテゴリ別売上の月次推移」を分析する場合、まず日別集計を行い、次に月別集計、最後に推移計算といった複数段階の処理が必要になります。MapReduceでは各段階が独立したジョブとなり、非効率的な処理になってしまいます。
  Apache Sparkによる改善
  これらの問題を解決するために開発されたのが分散データ処理フレームワークです。従来の制約を解消し、より柔軟で高性能な処理を可能にします。ここではApache Sparkについて主について概要を語ります。Sparkは2009年にカリフォルニア大学バークレー校のAMPLabで開発されました。

  Sparkの抽象データ構造
  Sparkの核となるデータ構造はRDD（Resilient Distributed Dataset）です。RDDは、クラスター内の複数ノードで並列処理可能な、分散されたオブジェクトのコレクションです。耐障害性を持ち、障害が発生しても自動的に復旧して処理を継続できます。
  Sparkフレームワーク活用者は、各種プログラミング言語でメモリ上のコレクションを扱うかのように並列データ処理を記述できる点が優れています。
  RDDは外部ストレージ（HDFSやS3など）からデータを読み込むことで作成するか、既存のRDDに対してmap、filter、reduceなどの操作を適用して変換することで作成できます。下にコード例を書きますが、PythonのPandasを活用したことがある方であれば馴染みやすいコードです。

  インメモリ処理による高速化
  Sparkでは、中間結果をメモリ上に保持することで、ディスクI/Oを大幅に削減します。特に反復処理が多いワークロードでは、同じデータセットを何度もメモリから読み出せるため、MapReduceと比較して10倍から100倍の性能向上を実現できます。
  例えば、機械学習アルゴリズムの学習処理では、同じ訓練データに対して数百回の反復計算を行います。メモリ上でデータを保持することで、各反復でのデータ読み込み時間を大幅に短縮できます。
  柔軟な処理モデル
  固定的な2段階処理の制約を解消し、任意の段数の変換処理を組み合わせることができます。開発者は「map」「filter」「groupBy」「join」などの多様なオペレーション（操作）を自由に組み合わせて、複雑なデータ処理パイプラインを構築できます。
  これにより、複数のMapReduceジョブが必要だった処理を、単一のワークフローとして実行できるようになります。
  バッチ処理がマッチするユースケース
  基本的に大量のデータを効率的に処理して正確な集計結果や分析結果を得たいというユースケースで有効です。リアルタイム性よりも、処理の正確性、効率性、コスト最適化を重視する場面で力を発揮します。

1. ECサイトの日次売上レポートと月次分析レポート
   ECサイトでは、1日の終わりに全注文データを集計して売上レポートを生成し、月末には詳細な分析レポートを作成する必要があります。数百万件の注文データから、商品別売上、カテゴリ別売上、地域別売上、顧客セグメント別分析などの複雑な集計処理を実行します。
   この場合、夜間のバッチ処理で1日分のデータをまとめて処理することで、高い処理効率を実現できます。MapReduceパターンを使用して、注文データを商品カテゴリやリージョンで分割し、並列処理で集計計算を実行します。また、過去のデータとの比較分析や、複数のデータソース（在庫データ、顧客データ、マーケティングデータ）を結合した高度な分析も行います。

大量のイベントデータをストリーム処理する
大量のイベントデータをリアルタイムで処理する必要があるシステムでは、ストリーム処理を活用することが一般的です。例えば、ユーザーのクリック行動を追跡してリアルタイムで分析したり、金融取引の不正検知を行ったりする場合、従来のバッチ処理では処理の遅延が発生してしまいます。このような状況で、ストリーム処理の設計パターンを理解することは非常に重要です。基本的にはデータエンジニアが専門で対応する分野ではありますが、広告クリックシステムの設計などで活用されることがあるため、システムデザイン面接で設計パターンの候補として知っておくのは大切です。
ストリーム処理とバッチ処理の違い
バッチ処理は、蓄積されたデータを一定の間隔で一括処理する手法です。一方、ストリーム処理は、データが生成されるとほぼリアルタイムで連続的に処理する手法です。
バッチ処理の特徴は、高いスループットと処理の確実性にあります。大量のデータを効率的に処理でき、処理が失敗した場合でも簡単にリトライできます。また、処理結果は毎回同じになるため、データの一貫性が保たれます。しかし、処理結果が得られるまでに時間がかかるため、リアルタイム性が求められる用途には適していません。
ストリーム処理の特徴は、低レイテンシとリアルタイム性にあります。データが生成されるとすぐに処理されるため、即座に結果を得ることができます。しかし、連続的なデータの流れを処理するため、状態管理が複雑になり、システムの実装が困難になります。
旧LINEのData Platform室ではApache Flinkによるストリーム処理のパイプラインを開発・運用しています。ストリーム処理されたデータはElasticsearchやHDFSに転送されています。

そもそもストリームとは？
ストリームとは、時間の経過とともに継続的に生成されるデータの流れを指します。従来のバッチ処理で扱う静的なデータセットとは異なり、ストリームは理論的に無限に続くデータシーケンスです。
ストリームの各データ要素は「イベント」と呼ばれ、特定の時点で発生した出来事を表します。ユーザーのクリック、センサーからの測定値、システムログエントリなど、様々な種類のイベントがストリームとして扱われます。ストリームデータの特徴は、データが連続的に到着し、全体のデータサイズが事前に分からないことです。そのため、従来のバッチ処理とは異なる処理パラダイムが必要になります。
ウィンドウ
ストリーム処理では、無限に続くデータストリームを有限の単位で処理するために「ウィンドウ」という概念を使用します。ウィンドウは、特定の時間範囲やイベント数に基づいてストリームを分割し、その範囲内でのみ集計や分析を行います。 Flinkのドキュメントも分かりやすいので合わせて見ておくとよいです。

ストリーム処理の時間の概念
ストリーム処理では時間の概念がいくつかあり、集計などの処理をどの時間をベースにして実行するのか、意識する必要があります。
Event Timeは、ストリーム処理においてイベントが実際に発生した時刻を指します。この時刻は通常、イベントデータ自体に含まれており、イベントの自然な順序を表します。Event Timeは、時系列分析やパターン検出において重要な基準となります。
Processing Timeは、ストリーム処理システムがイベントを実際に処理する時刻を指します。ネットワークの遅延、システムの負荷、一時的な障害などにより、Processing TimeはEvent Timeから遅れることがあります。
例えば、IoTセンサーからのデータを処理する場合、センサーが測定した時刻（Event Time）と、データがクラウドで処理された時刻（Processing Time）に差が生じることがあります。なお、別にIngestion timeもあるがあまり使用用途としてはありません。
Watermark
ストリーム処理システムにやってくるデータの順序はネットワーク遅延などの要因により通常期待される順序になっていない場合がある。これにより集計処理などで期待した通りの結果にならない場合があります。これを解決する手法としてWatermarkがあります。
Watermarkの仕組みは、ストリーム処理システムがいつウィンドウを閉じて集計結果を出力するかを決定するために使用されます。システムは、Watermarkが通過した時点で、そのウィンドウ内のデータが全てそろっていると判断します。
Watermarkの設定は、データの遅延特性とビジネス要件のバランスを取る必要があります。厳しすぎるWatermarkは遅延データを見逃す可能性があり、緩すぎるWatermarkは処理結果の出力を遅らせます。
例えば、以下のW(11)はtimestampが11以下イベントは来ないと見なします。Watermarkで定めたtimestampのデータが来た場合は無視されます。

ストリーム処理がマッチするユースケース
基本的にある一定時間（ウィンドウ）内に起きたイベントを集計したデータがリアルタイムで欲しいというユースケースで有効です。

1. 過去10分間の広告のクリックをリアルタイム集計
   オンライン広告システムでは、広告のクリック率やコンバージョン率をリアルタイムで監視する必要があります。過去10分間のクリック数を継続的に集計し、広告の効果を即座に評価できるシステムが求められます。
   この場合、10分間のSliding Windowを使用して、広告IDごとのクリック数を集計します。新しいクリックイベントが発生するたびに、過去10分間のクリック数が更新され、リアルタイムで広告の効果を把握できます。
   バッチ処理では、例えば1時間ごとにクリック数を集計することになり、広告の効果が悪い場合の対応が遅れてしまいます。ストリーム処理により、即座の意思決定と広告配信の最適化が可能になります。
2. 不正検知「5分以内に3回以上ログイン失敗したら警告」
   セキュリティシステムでは、不正アクセスを早期に検知するために、ログイン失敗のパターンを監視する必要があります。同一IPアドレスまたは同一ユーザーアカウントから5分以内に3回以上のログイン失敗があった場合、不正アクセスの可能性が高いと判断してアラートを発生させます。
   この場合、5分間のSession Windowを使用して、IPアドレスやユーザーIDごとのログイン失敗回数を監視します。閾値を超えた場合、即座にアラートを発生させ、該当するアクセスをブロックします。
   実際の例だと、JPMorganChaseは1日10兆ドルを超える決済取引の不正検知にリアルタイムのFeature EngineeringにFlinkを活用しています。

3. 検索インデックスを更新する
   Xのようなソーシャルメディアプラットフォームでは、話題の投稿に対して短時間で数万から数十万のいいねが発生し、これらのリアクション数を検索ランキングに反映させる必要があります。このような大量のリクエストを効率的に処理しながら、検索インデックスをリアルタイムで更新することが重要です。
   大量のリクエストが発生する際の典型的な問題は、データベースや検索エンジンへの負荷集中です。ストリーム処理では、リクエストを時間的に分散させることで、バックエンドシステムの過負荷を防ぐことができます。例えば、1秒間に1万件のリクエストが発生した場合でも、ストリーム処理により適切なレートで処理を実行し、システムの安定性を保つことができます。

アナリティクス用のデータベースを活用する
システムデザインにおいて、アナリティクス専用のデータベースを活用する場面としては主に広告クリック集計システムやWebサイトのページビューの集計システムなどで用いられることが多いです。
OLTPとOLAPの違い
データベースシステムは、その用途によって大きく2つのカテゴリに分類されます。OLTP（Online Transaction Processing）とOLAP（Online Analytical Processing）です。OLTPは我々が普段使用しているようなデータベースで、トランザクションの機能、データの一貫性などの機能が提供されています。
OLAP（Online Analytical Processing）とは？
OLAPは、大量のデータを分析するためのデータベースシステムです。過去のデータを集計・分析し、ビジネス上の意思決定を支援する役割を果たします。
OLAPの特徴は以下の通りです。
大規模な読み取り中心の処理: 膨大な量のデータを読み取って集計や分析を行いますが、データの更新頻度は低くなっています。
非正規化されたデータ構造: 分析に適した形にデータを事前に整理し、冗長性を許容してでも検索性能を優先します。
高スループット: 大量のデータを効率的に処理するため、並列処理や最適化された集計機能を提供します。
複雑なクエリへの対応: 多次元での集計やフィルタリングなど、複雑な分析クエリを効率的に実行できます。
例えば、売上レポートの作成、顧客行動の分析、市場トレンドの把握などがOLAPの典型的な用途です。
なぜOLAPデータベースは集計に向いているのか？
OLAPデータベースが集計処理に優れている理由は、その設計思想とアーキテクチャにあります。従来のOLTPデータベースでは効率的に処理できない大規模な集計処理を、以下のような仕組みで実現しています。
カラム指向ストレージ
OLAPデータベースの多くは、カラム指向（列指向）のストレージを採用しています。これは、テーブルのデータを行単位ではなく列単位で保存する方式です。
従来のOLTPデータベースでは行指向ストレージが使われており、1つのレコードに含まれる全ての項目が連続してディスクに保存されます。しかし、分析処理では特定の列のデータのみを集計することが多いため、この方式では不要なデータまで読み取ってしまい効率が悪くなります。
カラム指向ストレージでは、同一列のデータが連続して保存されるため、集計に必要な列のみを効率的に読み取れます。また、同じ種類のデータが連続しているため、圧縮効率も向上し、ストレージ容量とI/O処理の両方で優位性があります。
例えば、売上データの月別集計を行う場合、日付列と売上金額の列のみを読み取れば十分で、商品名や顧客情報といった他の列を読み取る必要がありません。
