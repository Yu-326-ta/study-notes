データベーストランザクション
銀行アプリやECサイト、SaaSの管理画面など、複数のユーザーが同時に同じデータにアクセスする状況は日常茶飯事です。そのような同時リクエスト（Concurrency）が発生する中で、データの一貫性をどう守るか？が、システム設計において非常に重要なポイントとなります
トランザクションとは？
まず前提となるのがトランザクションの考え方です。
例えば、AさんがBさんに1万円を送金する場合、以下の2つの処理を行います。
Aさんの口座残高を1万円減らす
Bさんの口座残高を1万円増やす
この2つの処理はどちらか一方だけが成功してしまうとデータの一貫性がなくなってしまうため、「両方まとめて一つの単位」として扱う必要があります。それがトランザクションです。
トランザクションには4つの性質（ACID特性）があり、特にここでは「Isolation（独立性）」が鍵になります。
トランザクション分離レベル
データベースでは、同時に走る複数のトランザクションが互いの処理にどれだけ影響しないようにするかを「分離レベル（Isolation Level）」で調整できます。
例えば、Aさんの口座残高を更新するトランザクションが実行している時に、他の処理がその口座残高を読み取ったら困りますよね？これを制御するのが分離レベルです。
代表的な4つの分離レベルを簡単に紹介します：
READ UNCOMMITTED: 他のトランザクションの未コミットのデータも見えてしまう（Dirty Read）。一貫性は最も低い
READ COMMITTED: 他のトランザクションのコミット済みのデータは読める。OracleやPostgreSQLのデフォルト
REPEATABLE READ: 同じトランザクション中の読み取り結果が常に同じになる（Fuzzy Readを防ぐ）。他のトランザクションでコミットされていても影響はない。MySQLのInnoDBのデフォルト
SERIALIZABLE: 完全に直列実行と同じ挙動。基本的に使用される事は少ない。
Aさんの口座から1万円を引き落とすとき、READ COMMITTEDやREPEATABLE READ以上であれば、途中で誰かがその残高を読み取って処理を誤るような事態を防ぎやすくなります

データベースロックを使って排他制御を行う
データベースロックとは、複数のトランザクションが同じデータに同時にアクセスしようとした際に、データの整合性を保つための仕組みです。
データベースの排他制御
データベースの排他制御とは、複数のトランザクションが同じリソースに対して競合する際に、適切な順序で処理を実行させる仕組みです。この仕組みにより、データの破損や不整合を防ぎ、システム全体の信頼性を確保します。
排他制御の基本的な考え方は、リソースへのアクセスを制限することです。あるトランザクションがデータを更新している間は、他のトランザクションがそのデータを変更できないように制御します。これにより、複数の処理が同時に実行されても、それぞれが独立して正しく動作することが保証されます。
排他制御には主にロックを使用した制御が用いられますで、データに対して明示的にロックをかけて他のアクセスを制限する方法です。
排他制御が必要な場面としては、例えば銀行口座から5千円を入金する処理と5000円を引き出す処理、同じ口座に同時に発生した場合、適切なロック機能がなければ以下のようにデータの不整合が発生してしまう可能性があります。
ちなみにこの、複数の処理が同時に行われた際に競合状態によって予期しない状態が引き起こされる問題をRace Condition（競合状態）と呼びます。

共有ロックと排他ロック
トランザクションの分離性を保つために、ロック（Lock）が活躍します。Aさんの口座情報を他のトランザクションから守るには、ロックを使ってアクセスを制限する必要があります。
ロックには主に次の2種類があります。例えば、Aさんの口座残高に共有ロックをかけていれば、他のユーザーがその残高を変更しようとしても処理がブロックされます。
こうして、更新処理が完了するまで他の処理を待たせることで整合性を守ることができます。
共有ロック（Shared Lock）
読み取り専用のロック。他のトランザクションは共有ロックをかける事ができますが、排他ロックをかける場合ブロックされます。つまり共有ロックをかける事で他のトランザクションで排他ロックをかけるようなUPDATE文などをデータの更新をブロックできます。SELECT ... FOR SHAREで明示的に特定の行（もしくは複数行）にロックをかける事ができます。これを利用する事で他のトランザクションに共有ロックをかけた行を読み取りしている事を示す事ができます。
BEGIN;

SELECT id, balance
FROM accounts
WHERE id = 1
FOR SHARE;
このクエリにより、id = 1 の口座情報に共有ロックがかかります。
他のトランザクションは同じ行に対して FOR SHARE であればロック取得可能（共存可能）
しかし、FOR UPDATE や UPDATE による排他ロック取得はブロックされる
この状態は、「残高の確認中に他の処理が勝手に更新してしまわないようにする」ケースで使えます。

排他ロック（Exclusive Lock）
書き込み用のロック。他のトランザクションが共有ロックや排他ロックを確保する操作をブロックします。UPDATE文やDELETE文などを実行する時には自動的に条件にマッチする特定の行（もしくは複数行）に排他ロックが取得されます。また明示的にSELECT ... FOR UPDATEで排他ロックをかける事もできます。データ更新の中に他のトランザクションが読み取り書き込みどちらもブロックしたい時に活用します。
BEGIN;

SELECT id, balance
FROM accounts
WHERE id = 1
FOR UPDATE;

UPDATE accounts
SET balance = balance - 1000
WHERE id = 1;

COMMIT;
この場合、FOR UPDATE によって id = 1 の口座に排他ロックがかかり、他のすべてのトランザクションがこの行に対して共有ロックと排他ロックをかけることはできない。

楽観ロックと悲観ロック
ロックのアプローチには2つの考え方があります。
悲観ロック（Pessimistic Lock）
悲観ロックはこれまで説明した手法です。「他の人がデータを更新してくるかもしれない」前提で、最初からロックをかけます。例えば、Aさんの口座に1万円を引き落とす処理では、最初に排他ロックをかけて他のアクセスを完全にブロックします。競合が多いシナリオでは悲観ロックを用いる事が多いです。
例えば、Aさんの口座（id = 1）から残高を引き出す処理で使うSQLです。このFOR UPDATEが、その行に排他ロックをかける部分です。これによりこのトランザクションが終了するまで、他のトランザクションはこの行を共有ロックも排他ロックもかける事はできません。つまり読み取りと書き込みをブロックできます。
START TRANSACTION;

SELECT id, balance
FROM accounts
WHERE id = 1
FOR UPDATE; -- 排他ロックがid=1の行にかかる

....

COMMIT; -- ロックが解除
楽観ロック（Optimistic Lock）
「ほとんど他と衝突しないだろう」と考えて、最後に整合性チェックだけ行います。例えば、「口座情報のversion番号」を保持しておいて、更新時に変更されていないかを確認します。つまりロックという言葉は使っていますが実際にはロックを使用していません。競合が少ないけど更新処理が重い場では楽観ロックが向いています。
具体的には次のようなSQLを発行します。このSQLは、「versionが5のときだけ更新する」という条件付き更新です。UPDATEのaffected rowsを確認し、更新されたのであれば1が返ってくるので成功です。しかし、0行の場合は他の誰かがすでに変更しているので再度トランザクションをやり直す必要があります。
UPDATE accounts
SET balance = balance - 1000,
version = version + 1
WHERE id = 1 AND version = 5; -- version=5の時だけ更新される

長時間のトランザクションへ分散共有ロックを活用する
データベースロックはトランザクションの整合性を守るために必要な技術です。では、データベーストランザクションのように短時間で終わるものではなく、長時間トランザクションが続く場合にはどうするべきでしょうか？
長時間のトランザクションが必要な場合
長時間トランザクションが続く例を考えてみましょう。例えば、映画館のチケット予約システムなどが良い例です。
ユーザーが「A列5番の席を予約」しようとしたとき、以下のような流れが発生します：
席の空き状況を確認
支払いページに遷移
クレジットカード入力
支払い処理
席を確定して予約完了
ユーザーが支払いが完了するまで他のユーザーがA列5番の席を予約することをブロックしなければなりません。
この一連の流れには数十分以上かかることもあります。ここでSELECT ... FOR UPDATE を使って、その席のレコードに排他ロックをかける方法はオススメできません。
ユーザーがクレカの入力を戸惑っているかもしれませんし、途中で離脱するかもしれません。その時にトランザクションが開いたまま = ロックが保持されたままでいるのは非常に危険です。
分散共有ロック
分散共有ロックを使うことで長時間のトランザクションにおける問題を解決できます。分散共有ロックの最大の利点は、有効期限（TTL）を設定できることです。
ユーザーが途中で離脱したり、アプリケーションがクラッシュしたりしても、一定時間後に自動的にロックが解除されます。これにより、他のユーザーが同じリソースにアクセスできるようになり、システム全体の可用性が保たれます。
また、分散共有ロックはデータベースのトランザクションとは独立して動作するため、長時間にわたる複雑な業務フローに対応できます。映画館の予約システムの例では、席の予約状態を分散ロックで管理し、実際のデータベース更新は最後の予約確定時に短時間のトランザクションで行うことができます。
例えば、Redisには「一定時間だけ有効なロック（TTL付き）」を実現する機能（SETNX）あります。これを利用する事でデータベースの特例のレコードへの長時間ロックを実現できます。
具体的には以下のように確保できます。
SET lock:seat:123 user456 NX EX 300
lock:seat:123 はロックのキー（席番号）
NX: 既にロックがある場合は取得しない（他ユーザーが予約中）
EX 300: 有効期限を5分（300秒）に設定

CAP定理と整合性モデル
システムデザイン面接において、分散システムのCAP定理と整合性モデルの理解することでシステムの非機能要件を満たす上で適切な選択を行うことができます。
CAP定理とは
CAP定理は分散システムにおいて同時に満たすことができない3つの特性を表した理論です。この定理は2000年にコンピュータ科学者のEric Brewerによって提唱され、分散システムの設計において重要な指針となっています。
CAPはそれぞれ以下の特性を表しています。
Consistency（一貫性）
すべてのノードで同じタイミングに同じデータが読み取れることを保証する特性です。あるノードでデータが更新された場合、他のすべてのノードでも即座に同じデータが反映される状態を指します。
Availability（可用性）
システム内の一部のノードに障害が発生した場合でも、残りのノードがリクエストに対して応答を返し続けることができる特性です。システム全体がダウンすることなく、常にサービスを提供できる状態を維持します。
Partition tolerance（分断耐性）
ネットワークの分断が発生した場合でも、システムが動作し続けることができる特性です。ネットワーク障害により一部のノード間で通信ができなくなっても、システム全体の動作を継続できる能力を指します。
CAP定理では、分散システムにおいてこれら3つの特性のうち最大でも2つまでしか同時に満たすことができないという制約です。つまり、CP（分断耐性と一貫性）、AP（分断耐性と可用性）、CA（一貫性と可用性） の3つの設計パターンのどれかを選択するということです。
しかしその後の議論により、CAという選択肢は現実的ではないことが明らかになってきました。なぜなら、分断耐性を持たないシステムは、ネットワーク分断が発生した瞬間に、一貫性か可用性のいずれかを失うことが避けられないからです。そのため、現代的なCAP定理の理解では「ネットワーク分断が発生した際、分散システムは一貫性か可用性のいずれかを選ばなければならない」という形で捉えられています。

BASE特性
BASE特性は分散システムにおける高可用性とスケーラビリティを重視するアプローチです。
Basically Available（基本的に可用）
常に利用可能であること。一部の障害があっても、システム全体としては利用可能である状態を目指します。﻿
Soft State（ソフトステート）
常に整合性を保っている必要はないという性質。一時的に整合性が取れていない状態があり得ます。﻿
Eventual Consistency（結果整合性）
直ちには整合性が保証されなくても、時間をかけて最終的に整合性が達成されれば良いという考え方
これは、トランザクション処理においてACID特性（原子性、一貫性、独立性、耐久性）を重視するRDBMSとは異なり、分散環境でのパフォーマンスと可用性を優先するために、ある程度の一貫性を犠牲にする考え方です。
CAP定理と非機能要件の関連性
システムデザインにおける非機能要件とCAP定理は密接に関係しています。非機能要件とは、システムがどのように動作するかを定義する品質属性のことで、パフォーマンス、可用性、スケーラビリティ、一貫性などが含まれます。
CAP定理を理解することで、システムの要求に応じて適切なトレードオフを判断できるようになります。重要なのは、ビジネス要件とユーザーの期待値を分析し、一貫性と可用性のどちらがより重要かを見極めることです。
この判断は以下の観点から行います。
データの正確性がビジネスクリティカルかどうかを評価します。金融システムのように、不正確な情報が重大な影響を与える場合は一貫性を重視する必要があります。
サービス停止によるビジネスインパクトを分析します。ソーシャルメディアや検索エンジンのように、短時間のサービス停止でも多くのユーザーに影響を与える場合は可用性を重視します。
ユーザーの利用パターンと期待値を考慮します。リアルタイム性を重視するサービスでは、多少のデータの不整合よりもサービス継続を優先することが多くあります。
可用性を重視するシステム設計
X（旧Twitter）のような大規模なソーシャルメディアサービスは、可用性を一貫性よりも重視する典型例です。これらのサービスでは、ユーザーが常にアクセスでき、投稿や閲覧ができることが最優先事項となります。
可用性を重視するサービスでは結果整合性モデルを採用することが多く、必ずしも最新の値を毎回取得できることを保証しません。
大規模なWebサービスにおいて可用性が重視される理由は複数あります。ユーザー体験の観点から、サービスが利用できないことは直接的な機会損失につながります。数百万から数億のユーザーを抱えるサービスでは、短時間のダウンタイムでも莫大な影響が生じます。
また、ソーシャルメディアの特性上、投稿の順序やタイムラインに多少の遅延や不整合があっても、ユーザーの体験に致命的な影響を与えることは稀です。例えば、「いいね」の数が一時的に異なって表示されたり、投稿の表示順序に多少のばらつきがあったりしても、サービス自体が利用できなくなるよりもはるかに受け入れられやすいものです。
ネットワーク分断が発生した場合の対応も重要な設計要素です。地理的に分散した複数のデータセンター間で通信障害が発生しても、各地域のユーザーは引き続きサービスを利用できるような仕組みが必要です。この場合、一時的にデータの不整合が発生する可能性はありますが、障害が復旧した後に最終的な整合性を保つことで問題を解決します。
一貫性を重視するシステム設計
銀行の口座管理システムは、一貫性を可用性よりも重視する代表的な例です。金融システムにおいては、データの正確性と整合性が何よりも重要であり、不正確な残高表示や重複した取引処理は重大な問題を引き起こします。
一貫性を重視するシステムでは結果整合性モデルではなく強整合性モデルを採用します。これは常に最新のデータが取得できることを保証します。
銀行システムで一貫性が最重要視される理由は明確です。顧客がどの支店のATMを利用しても、オンラインバンキングでアクセスしても、モバイルアプリで確認しても、常に同じ正確な残高が表示される必要があります。また、取引の処理において、二重支払いや残高不足時の不正な引き落としは絶対に防がなければなりません。
このようなシステムでは、ネットワーク分断が発生した場合、一貫性を保つためにサービスの一部を停止することが選択されます。例えば、メインデータセンターとバックアップデータセンター間の通信が切断された場合、データの整合性を保証できない状況では、残高の閲覧だけを許可し、新しい取引の処理を一時的に停止、通信が復旧してからサービスを再開します。
金融システムの設計では、ACID特性（Atomicity、Consistency、Isolation、Durability）を満たすトランザクション処理が必須となります。これにより、複数の操作を一つの不可分な単位として扱い、すべての操作が成功するか、すべてが失敗するかのどちらかの結果のみを許可します。

マイクロサービス間のトランザクションの整合性を維持する
分散システムにおけるトランザクション整合性の維持は、現代のマイクロサービス全盛期の時代においてシステム全体の信頼性と可用性を保証するための根幹ともいえる課題です。各コンポーネントが独立して動作し、ネットワークを介して連携する中で、一貫性を確保するための技術やアーキテクチャは、実際の運用現場で直面する現実的な問題となっています。
ここでは、まず分散システムにおけるトランザクションの基本と、Sagaパターンを紹介し、具体的な事例を通じてどのようにマイクロサービス間のデータの整合性を維持していくか説明していきます。
Sagaパターン
Sagaパターンは、分散トランザクションの整合性を管理するためのパターンであり、一連のローカルトランザクションの集まりとして定義されます。それぞれのローカルトランザクションは独立してコミットされ、失敗時には補償トランザクション（補償操作）を呼び出して元の状態に戻すことを目的としています。

Sagaパターンには、以下の2つの主要なアプローチがあります。
コレオグラフィ（Choreography）
コレオグラフィ方式では、各サービスが独立してトランザクションを実行し、イベント駆動で次のサービスに通知します。各サービスは「成功」「失敗」といったイベントを発生させ、他のサービスがそのイベントをリッスンして処理を進めます。
メリット:
サービスが疎結合であり、変更に強い
中央集権的な管理が不要
デメリット:
流れが複雑になりやすく、異常時のトラブルシューティングが難しい
イベントが多くなりすぎると追跡が困難

オーケストレーション（Orchestration）

デメリット:

メッセージキューとデータベースの書き込みを同一のトランザクション内で実行する
データベースへの書き込みと同時にメッセージキューへの送信を行う必要がある場面は数多く存在します。例えば、ECサイトで注文処理を行う際に、注文データをデータベースに保存すると同時に、在庫管理システムや配送システムに通知を送る必要があります。
しかし、この一見単純な処理には重大な問題が潜んでいます。データベースの書き込みとメッセージキューへの送信が別々のシステムである以上、一方が成功して他方が失敗する可能性があります。この問題を解決するのが、本章で解説するTransactional Outboxパターンです。
データベースとメッセージキュー間の一貫性の問題
分散システムにおいて、複数の独立したシステム間でデータの一貫性を保つことは困難です。特に、データベースとメッセージキューという異なる性質を持つシステム間では、以下のような問題が発生します。
データベースへの書き込みが成功したにもかかわらず、メッセージキューへの送信が失敗した場合を考えてみましょう。この状況では、データベースには新しいデータが保存されているにもかかわらず、他のシステムはその変更を知ることができません。逆に、メッセージキューへの送信は成功したものの、データベースへの書き込みが失敗した場合、他のシステムは存在しないデータに基づいて処理を行うことになります。

Transactional Outboxパターンとは
パターンの概要
Transactional Outboxパターンは、データベースの書き込みとメッセージの送信を一つのローカルトランザクション内で実行することで、一貫性を保証するデザインパターンです。このパターンでは、メッセージを直接外部のメッセージキューに送信する代わりに、同じデータベース内の専用テーブル（Outboxテーブル）に書き込みます。
具体的には、ビジネスデータの更新とOutboxテーブルへのメッセージ挿入を同一のデータベーストランザクション内で実行します。この後、別のプロセスがOutboxテーブルを監視し、新しいメッセージを実際のメッセージキューに送信します。
このパターンにより、最終的にはデータベースの書き込み成功 + 少なくとも一回以上はメッセージキューへ送信の成功を保証することができます。
パターンの動作フロー
Transactional Outboxパターンの動作は、以下の段階に分けることができます。
まず、ビジネスロジックの実行段階では、アプリケーションがビジネスデータの更新とOutboxテーブルへのメッセージ挿入を同一のトランザクション内で実行します。このトランザクションがコミットされると、データの更新とメッセージの記録が原子的に実行されることが保証されます。
次に、メッセージ送信段階では、Message Relayと呼ばれる別のコンポーネントがOutboxテーブルを定期的にポーリングまたは変更を監視し、新しいメッセージを発見すると実際のメッセージキューに送信します。送信が成功すると、該当するOutboxレコードを削除または処理済みとしてマークします。
notion image
Transactional Outboxパターンのデメリット
Message Relayのコンポーネントが間に挟むので余計な遅延が発生するためパフォーマンスが悪化する点、OutboxテーブルやMessage Relayといった追加のコンポーネントがあり複雑性が増す点、大量の書き込みがある場合はMessage Relayのスケーラビリティを考えなければならない点があります。これらのデメリットが無視できるほど、データベースとメッセージキュー間の一貫性の問題を解決したいときに役に立つパターンです。
実装方法
Outboxテーブルの設計
Outboxテーブルには、メッセージの送信に必要な情報を格納します。典型的なスキーマには以下のフィールドが含まれます。
ID: 各メッセージを一意に識別するための主キー
作成日時: メッセージが作成された日時
宛先: メッセージの送信先を識別する情報
ペイロード: 実際のメッセージ内容
処理状態: メッセージの処理状況を表すフラグ
再試行回数: 送信失敗時の再試行回数を記録
Message Relayの実装
Message Relayは、Outboxテーブルからメッセージキューへの送信を担当するコンポーネントです。このコンポーネントには、いくつかの実装方式があります。
ポーリング方式では、Message RelayがOutboxテーブルを定期的にクエリして新しいメッセージを発見します。この方式はシンプルで実装しやすいですが、ポーリング間隔とリアルタイム性のトレードオフがあります。
変更データキャプチャ方式では、データベースの変更ログを監視してOutboxテーブルの変更をリアルタイムで検出します。この方式はより効率的ですが、データベース固有の機能に依存するため実装が複雑になります。
重複送信の対処
Transactional Outboxパターンでは、システム障害時の再試行により同じメッセージが複数回送信される可能性があります。これを防ぐため、冪等性を確保する仕組みが必要です。
一般的なアプローチとして、各メッセージに一意なIDを付与し、受信側でこのIDを記録することで重複を検出する方法があります。また、メッセージの内容自体が冪等になるよう設計することも重要です。
障害時の対応
Transactional Outboxパターンにおいて、Message Relayやメッセージキューの障害は避けられません。Outboxテーブルに書き込みが成功しても結局これらのサービスに障害に対する適切な対応策を用意する必要があります。
メッセージキューへの送信が継続的に失敗する場合は、指数バックオフを用いたリトライロジックを実装します。一定回数のリトライも失敗する場合は、管理者への通知やDLQへの移動を行います。
また、Message Relayプロセス自体が停止した場合の復旧機構も重要です。プロセスの健全性監視とオートリスタート機能を実装することで、サービスの可用性を保つことができます。

コーディネーションサービスを活用した分散システムの管理
コーディネーションサービスは、分散システムにおける複雑な調整問題を解決するための基盤技術として、多くの大規模システムで採用されています。リーダー選出、サービスディスカバリーなどの機能を提供し、開発者が複雑な分散合意アルゴリズム（RaftやPaxos）を実装することなく、信頼性の高いシステムを構築できるようにします。
コーディネーションサービスとは？
コーディネーションサービスとは、分散システム内の複数のノード（システム）が協調して動作するために必要な調整機能を提供する専用のサービスです。例えば、Apache ZooKeeper、etcd、Consulなどが代表的なコーディネーションサービスとして知られています。非常に分かりにくいサービスですが、分散システムのための中央集権でメタデータを管理するキーバリューストアと呼んだ方が分かりやすいかもしれません。
基本的な特徴
コーディネーションサービスは、以下のような特徴を持ちます。
一貫性のあるメタデータストレージを提供します。全ての参加ノードが同じ情報を共有できるよう、強い一貫性（Strong Consistency）を保証したデータストレージ機能を備えています。これにより、システム全体で統一された情報を維持できます。例えば、ZooKeeperではZAB（ZooKeeper Atomic Broadcast）プロトコルを使用してリーダーノードが全ての書き込み処理を統制し、過半数のノードが更新を確認してから変更を確定することで、全てのノードが同じデータを持つことを保証します。
リアルタイム通知機能により、システムの状態変化を即座に関係するノードに伝達します。従来のポーリング方式と異なり、変更が発生した瞬間に通知を受け取ることができるため、システムの応答性が大幅に向上します。例えば、ZooKeeperではWatchメカニズムを提供しており、クライアントが特定のZNodeに対してWatchを設定すると、そのZNodeのデータ変更や子ノードの追加・削除が発生した際に自動的にイベント通知を受け取ることができます。
高可用性設計により、サービス自体が単一障害点になることを防ぎます。複数のノードで構成されたクラスター構成を採用し、一部のノードが停止してもサービス全体は継続して動作します。例えば、ZooKeeperクラスタでは通常3台、5台、7台の奇数台でアンサンブルを構成し、過半数のノードが稼働していれば正常にサービスを提供でき、例えば5台構成であれば2台のノード障害まで耐えることができます。
データモデル
コーディネーションサービスは通常、階層的なキーバリューのデータモデルを採用しています。各データノードには以下のような特性があります。今回はZooKeeperを例に説明していきます。
パス指定によるアクセスが可能で、ファイルシステムのパスのように「/アプリケーション名/サーバー群/サーバー1」といった形式でデータを特定できます。
あくまでメタデータの管理するためのデータストアであり小さなデータサイズが想定されており、大量のアプリケーションデータの保存には向いていません。例えば、ZooKeeperではノード（znode）のデータサイズ、デフォルトマックスは1MB。
ノードタイプの選択により、用途に応じた動作を実現できます。永続的なノード（persistent node）は明示的に削除されるまで存在し続け、一時的なノード（ephemeral node）はクライアントの接続が切れると自動的に削除されます。

コーディネーションサービスは何故必要？

しかし、ユーザー数が増加すると、単一サーバーでは処理能力に限界が生じます。そこで、2台目のサーバーを追加することになります。
分散システムで発生する問題
2台のサーバー構成になった瞬間、問題が発生します。SakamotoがServer1に接続し、AbeがServer2に接続している場合、SakamotoからAbeへのメッセージ配信にはサーバー間の通信が必要になります。

1. サービスディスカバリーの問題
   サービスディスカバリーの問題が最初に現れます。Chat Server1は、Abeがどのサーバーに接続されているかを知る必要があります。この情報をどこで管理し、どのように最新の状態を維持するかが課題となります。2台のサーバー構成の場合はまだよいですが、これがN個になった場合どこにAbeが接続されているかどのように知ればよいでしょうか？

2. データベースを使った解決策の限界
   データベースを使った解決策の限界を考えてみましょう。ユーザーとサーバーのマッピング情報をデータベースに保存する方法は一見よい解決策見えます。小規模なうちはこれでも問題ないでしょう。しかし、データベースが単一障害点となり、停止すると全体のチャットシステムが機能しなくなります。
   また、全てのメッセージ配信でデータベースへの問い合わせが必要となり、遅延が発生します。さらに、ユーザー数が数百万人規模になると、データベースへの負荷が集中してボトルネックとなります。
   notion image
3. サーバー間直接通信の課題
   各サーバーが他の全てのサーバーと直接通信する方式では、例えば、サーバー数をnとすると、Chat Server 1はどこにユーザーが存在するか分からないので毎回n-1回の通信が発生します。これはPub/Subサービスを利用すると比較的簡単に実装できますが、これはn(n-1)/2 ⇒ n²個の接続が発生するので、n数によっては膨大な接続数を管理することとなります。

4. 障害検知の困難さ
   Chat Server2がクラッシュした場合、そのサーバーは他のサーバーに停止を通知できません。ハートビート機能を実装しても、ネットワーク分断が発生した場合、Chat Server2が実際には稼働しているにも関わらず、他のサーバーから停止していると誤認される可能性があります。

分散システムでユニークIDを設計する
システム設計において、各エンティティを一意に識別するためのユニークIDの選択は重要な設計判断です。IDの設計によってシステムの性能、スケーラビリティ、可用性、そしてセキュリティが大きく左右されます。本章では、主要なID生成方式の特徴とトレードオフを体系的に解説し、要件に応じた最適な選択ができるようになることを目指します。
なぜ分散システムで連番のユニークIDの設計が重要なのか
分散システムでは、単純な連番のユニークIDを使用するには大きな課題があります。複数のサーバーやデータベースインスタンス間でIDの一意性を保証しつつ、単一障害点を防ぎ、高いスループットを維持する必要があります。適切なID設計により、以下の要件を満たすことができます。
AUTO_INCREMENT（自動採番）
AUTO_INCREMENTは最もシンプルで理解しやすいID生成方式です。データベースが自動的に1から始まる連続した整数を割り当てます。
この方式の最大の利点は、生成されるIDが時系列順に厳密にソートされることMySQLなど主要なRDBでデフォルトでサポートされていることです。また、サイズが小さく（4バイトまたは8バイト）、インデックスの効率が非常に高いため、検索や挿入性能が優秀です。
しかし、分散システムにおいては深刻な制約があります。複数のデータベースインスタンス間でIDの重複を避けるには、中央集権的なID発行システムが必要となり、これが単一障害点となってしまいます。また、連番のため外部から推測されやすく、システムの規模や成長率が暴露されるリスクもあります。
例えば、URLに含まれるIDを変更することで他のユーザーのデータにアクセスを試みる攻撃が可能になります。
UUIDv4
UUIDv4は128ビットの完全にランダムな識別子です（正確にはVersionやVariantを除いた122ビットがランダム）。

+----------+----------+----------+----------+
| Random | Version | Random | Variant |
| 48bits | 4bits | 12bits | 2bits |
+----------+----------+----------+----------+
| Random 62bits |
+-------------------------------------------+
UUIDv4の最大の利点は、分散環境で中央調整なしに一意なIDを生成できることです。衝突確率は天文学的に低く、実用上は無視できます。また、完全にランダムなため推測が不可能で、セキュリティの観点で優秀です。

ULID（Universal Lexicographically Sortable Identifier）
ULIDは2016に仕様が公開された、時刻順序性を重視した128ビットの識別子で、Base32エンコーディングによる26文字の文字列表現が特徴です。

UUIDv7
UUIDv7は2024年に標準化された新しい形式で、ULIDの良さを引き継いだ時刻順序性とランダム性を両立させた設計です。
+----------+----------+----------+----------+
| Timestamp| Version | Random A | Variant |
| 48bits | 4bits | 12bits | 2bits |
+----------+----------+----------+----------+
| Random B (62bits) |
+-------------------------------------------+
UUIDv7は既存のUUID標準との互換性を保ちながら、ULIDの時間順序性の利点を取り入れた画期的な設計です。48ビットのUNIXエポック（ミリ秒精度）を最上位ビットに配置することで、生成時刻順での自然なソートが可能になり、データベースのB+treeインデックスでの挿入性能が大幅に改善されます。
ULIDと比較して、UUIDv7は既存のUUIDエコシステムとの完全な互換性を持つため、多くのライブラリやツールでそのまま使用できます。また、標準的なUUID形式（8-4-4-4-12のハイフン区切り）を採用しているため、既存システムへの移行が容易です。

Snowflake ID
Snowflake IDはTwitter（現X）が2010年6月1日に発表した64ビットの時刻ベース識別子で、大規模分散システムでの高スループットを実現します。
+----------+----------+----------+----------+
| Sign(1) | Time | Machine | Sequence |
| 0 | 41bits | 10bits | 12bits |
+----------+----------+----------+----------+
Snowflake IDは41ビットのミリ秒タイムスタンプ（UNIXエポックではなくカスタムエポック）、10ビットのマシン識別子、12ビットのミリ秒内カウンタを結合した64ビットの整数として表されます。1のマシンあたりで1ミリ秒に最大 4096個、すなわち毎秒409万のIDを発行できます。整数型のため、データベースやプログラムでの扱いが単純であるという点はUUIDやULIDにはないメリットの一つです。
しかし、正確な時刻同期が必要で、時刻の逆行が発生するとID生成が停止する可能性があります。また、マシンIDが固定のため、システム構成に関する情報が推測される可能性があります。
例えば、Snowflake IDはInstagram、Mastodon、DiscordでもSnowflake IDの改訂版が使用されており、さらにその後に発表された一意でソート可能な分散システムにおけるIDの設計に関して大きな影響を与えました。
どのように採番するか？
基本的な採番アルゴリズム
ID生成の基本的な流れは以下の通りです。現在時刻をミリ秒単位で取得し、カスタムエポック（例えば2025年8月1日）からの経過時間を計算します。事前に割り当てられたマシンIDを設定し、同一ミリ秒内での生成回数に応じてシーケンス番号を管理します。
def generateSnowflakeId():
timestamp = getCurrentTimeMillis() - CUSTOM_EPOCH
machineId = getMachineId()
sequenceNumber = getNextSequenceNumber()

    return (timestamp << 22) | (machineId << 12) | sequenceNumber

ファイルをオブジェクトストレージにアップロードする
システムデザインでは、画像、動画などのファイルの保存と配信は重要な要素となります。InstagramやXのようなSNSサービスでは、ユーザーがアップロードした画像や動画を保存する必要があります。また、DropboxやGoogle Driveのようなクラウドストレージサービスでは、ユーザーがアップロードしたファイルやドキュメントを安全に保存し、いつでもアクセスできるようにすることが重要です。
ここでは、非構造化データのファイルの保存を行うために必要な技術とインフラの設計について学んでいきましょう。
オブジェクトストレージ
システムデザインにおいて、オブジェクトストレージは主に画像、動画、大容量の非構造化データの保存に使用されます。例えば、Amazon S3やGoogle Cloud Storageはオブジェクトストレージのサービスとしてとても有名ですね。オブジェクトストレージは、データを「オブジェクト」として保存し、それぞれが実際のデータ、メタデータ、そしてユニークな識別子を含むという特徴を持っています。このストレージタイプは、従来のファイルシステムとは異なり、階層構造を持たないフラットなアドレス空間で管理されます。これにより、ディレクトリやフォルダの階層を扱う必要がなく、ストレージの増加に伴う複雑性が軽減されます。
オブジェクトストレージ以外の選択肢を検討する場合、まず代表的なものとしてリレーショナルデータベース（RDB）が挙げられます。しかし、RDBはトランザクション性や構造化データの整合性に優れている一方で、大規模なバイナリデータの保存には最適化されていません。特に画像や動画のような非構造化データは、ストレージサイズ、I/Oスループット、バックアップコスト、パフォーマンス面で多くの制約を受けるため、RDBに直接保存することは一般的に推奨されません。
ファイルストレージも一案ではありますが、分散構成におけるスケーラビリティ、堅牢性、可用性、コンテンツ配信ネットワーク（CDN）との統合といった観点で追加の設計が必要になります。
その点、オブジェクトストレージは設計の初期段階からスケーラブルな非構造化データの保存・配信を目的としており、現代のクラウドネイティブな要件に適した特性を備えています。そのため、画像や動画といった静的コンテンツの保存先として、今日のシステムデザインにおいてはオブジェクトストレージを第一の選択肢とし、他の方式を検討する必要性はほとんどないと言っても過言ではありません。

どのようにファイルをオブジェクトストレージに保存するのか？

セキュリティの面でも限定的なリソースへのアクセス許可しか与えないのでセキュアである

Pre-Signed URLなどの具体的な機能をシステムデザイン面接で使っていいのか？
一般的にシステムデザインでは固有名詞（製品名）を使わない形で設計を提案する場合が多いです。つまり面接中はS3ではなくオブジェクトストレージと言うべきであるという事です。ですが、共通化された機能に関しては発言してもよいと個人的に考えています。例えばPre-Signed URLはS3の機能だけではなくAzure Blob StorageにもShared Access Signatures (SAS)という同様の機能が存在します。
世の中クラウドネイティブなシステム設計が一般的になっている中では、Pre-Signed URLはベストプラクティスの一つですし、是非面接官に提案してみてください。もちろん、古典的なアップロード手法を提案する事は間違いではないですし、Pre-Signed URLが駄目だという面接官も世の中にはいるかもしれません。
最後に大事なのはコミュニケーションであり、ユースケースに応じた手札は多いほうがどのような状況になっても対応できるという事をお忘れなく。
