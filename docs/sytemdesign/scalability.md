垂直スケーリングと水平スケーリング
大規模なシステムデザインにおいて基本的には計算量視点とデータ量視点の両面でスケーラビリティは考えられます。つまり大量のリクエストをどのように処理するか？大量のデータをどのように保存すべきか？などの課題が議論の焦点になってきます。
垂直スケーリングと水平スケーリング
まず大事な概念として垂直スケーリング（スケールアップ）はあります。これは既存のサーバーのリソース（CPU、メモリ、ストレージなど）を増強することでシステムの処理能力を向上させる方法です。垂直スケーリングのメリットは、シンプルで実装が容易ですが、デメリットとしては、物理的なリソースの上限に制約されること、コストが高い、状況に応じて柔軟にスケーリングできないことが挙げられます。システムデザインの文脈ではあまり使われる事はないですが、次の水平スケーリング（スケールアウト）との比較で知っておけばよいです。
一方で水平スケーリング（スケールアウト）はサーバーの台数を増やし、負荷を分散させることでシステムの処理能力を向上させる方法です。水平スケーリングのメリットは、理論上は無限にスケール可能であること、単一障害点のリスクを低減できること、そしてコスト効率が良い場合が多いこと、状況に応じてスケーリングが可能なことが上げられます。

オートスケールの注意点
オートスケールの最も重要な制約として、仮想マシン（インスタンス）の起動時間による遅延があります。仮想マシンベースのオートスケールでは、インスタンスの起動から実際にリクエストを処理できる状態になるまで数分かかることが一般的で、この起動時間中は既存のインスタンスがすべての負荷を受け持つことになります
AWS公式としても注意喚起をしています。急激なスパイクには対応しきれないケースがあるのでそれを想定した設計をしなければなりません。

ロードバランサーでリクエストを負荷分散
一つのサーバーにリクエストが集中するとスケーラビリティの低下につながる可能性があります。ここでは大量のリクエストをどのように分散させるかの方法について学んでいきましょう。
ロードバランサー
ロードバランサー (Load Balancer) を用いると、クライアントからのリクエストを複数のサーバーへ振り分けることできます。ロードバランサーを導入する事でリクエストを負荷分散させることができます。
基本的にアプリケーションサーバーはHTTP通信を基盤としてステートレスで構築します。これは、ロードバランサーがどのアプリケーションサーバーにリクエストを送ったとしても正常に処理できなければ、スケールアウトが困難だからです。基本的にはセッション情報は外部のデータベースに保存することで、どのアプリケーションサーバーでも同じように処理できるようにします。

ロードバランサーのセッションアフィニティ
セッションアフィニティ（スティッキーセッション）は、特定のユーザーからのリクエストを常に同じアプリケーションサーバーに転送する仕組みです。これにより、アプリケーション側でセッション状態をローカルメモリに保持していても問題ありません。クライアントのIPアドレスやセッションCookieなどの識別情報を使って転送先サーバーを決定します。ユーザーが初回アクセス時に割り当てられたサーバーに、その後のリクエストも継続して送信します。

ロードバランサーの種類
L4ロードバランサー
L4ロードバランサーは、OSI参照モデルの第4層（トランスポート層）で動作し、IPアドレスとポート番号の情報を基にリクエストを振り分けます。TCPやUDPパケットの情報のみを参照するため、高速な処理が可能で大量のトラフィックを効率的に処理できます。
アプリケーションの内容を理解できないため、HTTPヘッダーやURLパスなどの詳細な情報に基づいた振り分けはできません。例えば、HAProxyのL4モード、Nginxのstreamモジュール、AWS Network Load Balancer（NLB）などがL4ロードバランサーの機能を提供しています。
L7ロードバランサー
L7ロードバランサーは、OSI参照モデルの第7層（アプリケーション層）で動作し、HTTPヘッダー、URLパス、クッキー、クエリパラメータなどのアプリケーション情報に紐づく詳細な情報を参照してリクエストを振り分けます。アプリケーションの内容を理解できるため、より柔軟で高度なルーティングが可能です。
特定のURLパスを持つリクエストを専用のサーバーに振り分けたり、ユーザーエージェントに基づいてモバイル向けサーバーとデスクトップ向けサーバーを使い分けたりできます。ただし、パケットの解析処理が必要なため、L4ロードバランサーと比較すると処理速度は劣る場合があります。例えば、HAProxyのL7モード、NginxのHTTPロードバランシング機能、AWS Application Load Balancer（ALB）などがあります。
DNSロードバランサー
DNSロードバランサーは、DNS（Domain Name System）の仕組みを利用してリクエストを分散させる手法です。クライアントがドメイン名を問い合わせた際に、複数のIPアドレスのうちどれかを返すことでアクセスを分散させます。
システムデザイン面接で有効な使い方として、地理的に分散したサーバー間での振り分けが得意です。これはクライアントのリクエストの送信元に地理的に最も近いサーバーへルーティングを行う機能です。
一方でDNSは結果整合性モデルのシステムなので、即座に振り分け先を変更できない場合があります。また、サーバーの実際の負荷状況を考慮した振り分けなどアプリケーションの状態を考慮した負荷分散は困難です。例えば、AWS Route53やCloudflare DNSなどのマネージドDNSサービスがDNSベースの負荷分散機能を提供しています。

from flask import Flask, jsonify
import mysql.connector
from mysql.connector import Error

app = Flask(**name**)

DB_CONFIG = {
'host': 'localhost',
'user': 'your_username',
'password': 'your_password',
'database': 'your_database'
}

def check_database_connection():
try:
connection = mysql.connector.connect(\*\*DB_CONFIG)
if connection.is_connected():
cursor = connection.cursor()
cursor.execute('SELECT 1')
cursor.fetchone()
cursor.close()
connection.close()
return True
else:
return False
except Error as e:
print(f"Database connection failed: {e}")
return False

@app.route('/health', methods=['GET'])
def health_check():
db_status = check_database_connection()  
 # DBの接続がOKであれば200、そうでなければ500を返す
http_status = 200 if db_status else 500
status = {
"status": "OK" if db_status else "FAIL"
}
return jsonify(status), http_status

if **name** == '**main**':
app.run(host='0.0.0.0', port=5000)

ロードバランサーの代表的なアルゴリズム
リクエストをどのように振り分けるのかロードバランサーにもアルゴリズムが複数存在します。代表的なものは
ラウンドロビン (Round Robin): 順番にサーバーへリクエストを振り分けていく、最もシンプルな手法。
最小コネクション数 (Least Connections): 現在のアクティブなコネクション数が最も少ないサーバーにリクエストを振り分ける。
最小レスポンス時間（Least Response Time）: 平均レスポンス時間が最も小さいサーバーにリクエストを振り分ける。
などあります。例えばAWSのELBデフォルトのバランシングアルゴリズムはラウンドロビンです。
全てまるっと覚える必要はなく、代表的なバランシングアルゴリズムをある程度想像するとよいでしょう。また、L7ロードバランサーであれば、HTTPヘッダー情報やURLからリクエストをよりスマートな方法で振り分け可能です。

マルチリーダー・リーダーレスレプリケーションによる書き込みクエリの分散
シングルリーダーレプリケーションでは書き込みクエリの分散ができない課題がありました。この問題を解決するために、マルチリーダーレプリケーションとリーダーレスレプリケーションという2つのアプローチがあります。
マルチリーダーレプリケーション
マルチリーダーレプリケーションは、複数のプライマリノード（リーダー）が存在し、それぞれが書き込みクエリを受け付けることができる仕組みです。各プライマリノードは書き込みを受け付けた後、その変更を他のプライマリノードとレプリカノードに伝播させます。
この仕組みにより書き込みクエリを複数のノードに分散させることが可能になり、書き込み性能の向上が期待できます。特に地理的に分散したシステムでは、各地域にプライマリノードを配置することで、ユーザーに近いノードへの書き込みによりレスポンス時間を短縮できます。

パーティショニングによるスケーラビリティの向上
データベースのパーティショニング（分割）することで大量のリクエストとデータに対してスケーラビリティを向上させる方法について話します。データベースのパーティショニングの利用箇所は数多くあり、それこそRDB、NoSQL、メッセージキュー、インメモリデータベースまで多岐にわたるます。いずれもパーティションキー（分割キー）による、データを分散することを目的としています。
パーティショニングで何故スケーラビリティが向上するのか？
パーティショニングがスケーラビリティを向上させる理由は、システムのボトルネックを分散させることにあります。単一のデータベースサーバーには物理的な制約があり、CPU処理能力、メモリ容量、ディスクI/O、ネットワーク帯域幅すべてに上限が存在します。これらの制約により、リクエスト数やデータ量が増加するとパフォーマンスが低下し、最終的にはサービスの停止につながる可能性があります。
パーティショニングを導入することで、これらの物理的な制約を複数のサーバーに分散できます。読み取りクエリだけでなく、書き込みクエリの負荷も複数のサーバーに分散されるため、単一のサーバーでは処理できない大量のトラフィックを処理することが可能になります。また、データを複数のサーバーに分散することで、一つのサーバーでは保存できない大容量のデータセットも扱えるようになります。
例えば、Kafkaでは各トピックを複数のパーティションに分割し、それぞれ異なるブローカーサーバーに配置することで、メッセージの書き込みと読み取りの負荷を分散します。単一ブローカーでは秒間数千メッセージしか処理できない場合でも、複数台のブローカーに分散することで秒間数万メッセージの処理が可能になります。また、新しいブローカーを追加してパーティションを再配分することで、トラフィック増加に応じてスケールアウトできます。
Cassandraでも同様に、データをパーティションキーでハッシュ化して複数のノードに分散配置します。各ノードが独立してクエリを処理するため、ノード数の増加に比例してシステム全体の処理能力が向上します。単一ノードでは数GB程度のデータしか効率的に処理できませんが、数百ノードのクラスターでは数TB以上のデータセットを高いパフォーマンスで処理できます。
パーティショニングの方法
何をパーティションキーにするべきか？
適切なパーティションキーの選択は、パーティショニングの成功を左右する最も重要な決定の一つです。理想的なパーティションキーは以下の特性を満たします。
まず、データが均等に分散されることが重要です。特定のパーティションにデータが集中すると、そのパーティションがボトルネックとなり、パーティショニングの効果が薄れてしまいます。データの分布が偏る原因として、時系列データでの日付や、人気の偏りがあるカテゴリIDなどが挙げられます。これらを避け、データが自然に均等に分布するキーを選択することが大切です。
次に、クエリパターンとの整合性を考慮する必要があります。アプリケーションが頻繁に実行するクエリで使用される条件と一致するパーティションキーを選ぶと、クエリを単一のパーティションで処理できるため、パフォーマンスが向上します。逆に、パーティションキーと異なる条件でクエリを実行すると、複数のパーティションを横断して検索する必要があり、パフォーマンスが低下します。
データの局所性（Data Locality）も重要な考慮点です。一緒にアクセスされることが多い関連データを同じパーティションに配置することで、取得時のパフォーマンスを大幅に向上させることができます。複数のテーブルや関連するレコードを横断してデータを取得する場合、それらが異なるパーティションに分散していると、ネットワーク通信やディスクI/Oが増加し、レスポンス時間が悪化します。関連データを同一パーティションに集約することで、単一のクエリで必要なデータを効率的に取得できます。例えば、SaaSアプリケーションの場合はテナントIDをパーティションキーとすることで関連データを全て同じパーティションに配置することができます。
どのようにパーティショニングするか？
レンジパーティションニング
レンジパーティションニングは、パーティションキーの値の範囲に基づいてデータを分割する手法です。日付、ID、価格など、順序がある値に対して有効です。データを連続する範囲に分割するため、範囲クエリ（特定の期間や価格帯での検索）に対して高いパフォーマンスを発揮します。
しかし、データの分布が不均等になりやすいという課題があります。特に時系列データでは、最新の時間範囲に書き込みが集中するホットスポット問題が発生する可能性があります。例えば、注文テーブルを注文日でレンジパーティショニングした場合、最新の日付のパーティションに書き込みが集中してしまいます。
ハッシュパーティショニング
ハッシュパーティショニングは、パーティションキーにハッシュ関数を適用して、その結果に基づいてデータを分割する手法です。ハッシュ関数の特性により、データが各パーティションに均等に分散されるため、負荷の偏りを避けることができます。どのような値の分布のデータでも比較的均等に分散できることが最大の利点です。
しかし、レンジクエリには適しておらず、特定の範囲の値を検索する場合は全パーティションを調べる必要があります。例えば、ユーザーIDをハッシュ化してユーザーデータを分散する場合、特定のユーザーのデータは効率的に取得できますが、ID範囲を指定した検索には不向きです。

リバランシング: データの分布が不均一であったり、特定のシャードに負荷が集中した場合、パーティショニングスキームを変更してデータを再分配する必要がありますが、これは簡単ではありません。何度もリバランスが起きると運用コストが高くなのでシャーディングを採用する際は慎重になりましょう。この問題をある程度解決できる手法として、分散データベースでよく使われているConsistent Hashingという手法があります。詳しくは

テーブル結合と非正規化: 異なるシャード間では当然テーブルの結合ができないので、慎重にシャードキーを選択する必要があります。シャード間で頻繁に結合しなければならない場合、そのもそも設計が間違っているか、テーブルの非正規化を考えて結合が起きないようにしましょう。
ホットスポット問題: 特定のキーに対するアクセスが集中すると、そのキーを含むパーティションの負荷が高くなります。人気のある商品情報や著名人のユーザー情報などがホットスポットになりやすいです。この問題に対処するには、シャードキーの設計の見直しが必要になります。
垂直分割
垂直分割では、特定の機能に関連するテーブルをそれぞれ別のサーバーに分割して保存します。これの嬉しい点としては現代のマイクロサービスとの相性が良い点です。各マイクロサービスが独立して特定の機能やデータを管理する構造がベストプラクティスなので、何も意識しなくてもこの分割方式を採用している事も多いです。また、各サービスが異なるデータベース技術を採用することも可能で、最適なデータストレージソリューションを選択できるという利点もあります。
例えば、下のようにUserテーブルとUser Photoテーブルは同じDBとテーブルで管理する事ができますが、テーブルを分割してさらに違うデータベースに置くようにします。

どのように垂直分割するか？
基本的には依存関係のない機能やデータのまとまり（ドメイン）ごとに分割するのが基本方針となります。ユーザー情報、商品情報、決済情報、通知情報など、ビジネス上の役割が異なり、相互に強い結合を持たない単位を見つけて、それぞれを独立したデータベースやサービスとして切り出します。
強い結合を保つ場合、分散トランザクションが必須となったり、結局アクセスが分散しておらずデータベースを分割した利点がなくなるからです。

Redisのハッシュスロットによるリバランシング問題の解決
従来のハッシュパーティショニングでは、ノード数が変更されるとハッシュ関数の結果が大幅に変わり、大量のデータを再配置する必要がありました。例えば、3台のサーバーから4台に増やす場合、ほぼすべてのデータが移動対象となってしまいます。
Redisクラスターはこの問題をハッシュスロットという概念で解決しています。16384個の固定されたハッシュスロットを用意し、各キーをスロット番号に変換してからノードに割り当てます。ノードの追加・削除時は、スロットの再割り当てだけで済むため、データ移動量を最小限に抑えることができます。
例えば、3ノードから4ノードに拡張する際は、各既存ノードから約1/4のスロットを新ノードに移動させるだけで、バランスの取れた分散を実現できます。これにより、大規模なデータセットでもサービス停止なしでスケーリングが可能になります。

パーティショニングが必要な理由
単一のメッセージキューサーバーには物理的な限界があります。メモリ容量、CPU処理能力、ネットワーク帯域幅、すべてに上限があるため、トラフィックが増加すると処理が追いつかなくなります。パーティショニングによって、これらの制約を複数のサーバーに分散させることで、システム全体の処理能力を向上させることができます。
例えば、Kafkaでは複数のブローカー（サーバー）でクラスターを構成し、各トピックを複数のパーティションに分割することで負荷を分散します。
もう一つの重要な理由は、メッセージの順序保証です。すべてのメッセージを1つのキューで処理すると、メッセージの処理順序は保たれますが、並列処理ができません。パーティショニングを使用することで、関連するメッセージを同じパーティションに配置し、パーティション内では順序を保ちながら、異なるパーティション間では並列処理を可能にします。
例えば、Kafkaでは同じキーを持つメッセージは必ず同じパーティションに送信され、そのパーティション内では送信された順序が保証されます。異なるパーティション間では順序は保証されませんが、各パーティションを異なるコンシューマーが並列で処理することで、全体的なスループットを向上させることができます。

ハッシュ値空間を固定長のリング（ハッシュリング）として扱い、各ノードはリング上のランダムな複数位置を担当します。データキーをハッシュして得た位置から時計回りに最初に現れるノードがそのキーを保持するようにします。ノードが追加・削除された時には、キーの移動は担当区間の前後ノードだけに限定されます。
例えば、S0~3のノードが以下のようにハッシュリング上に並んでいます。k0はS0、k1はS1、k2はS2、k3はS3が担当します。ここに新しく、S4を追加するとします。するとS4が担当するハッシュの領域はk0から時計回りにみてS0より近いので、データキーの再配置はk0がS0からS4 に移動します。その他のデータは移動の必要がないため、最低限のリバランスで済みます。

メッセージキューによるスケーラビリティの向上
メッセージキューは非同期処理の文脈でよく出てきますが、システム全体のスケーラビリティを向上させる上で大きく役割を果たします。
メッセージキューとスケーラビリティ
メッセージキューを活用することで、非同期処理のスケーラビリティ向上が容易になります。システムに大量のリクエストが流入した際、タスクをキューに蓄積し、複数のワーカープロセスが並列で処理することができます。
この仕組みの最大の利点は、処理能力をワーカーの数に比例して増やせることです。例えば、キューに蓄積されたタスクの量に応じて、ワーカーの数を動的に調整することで、負荷の変動に柔軟に対応できます。例えば、平常時は3台のワーカーで十分だった処理が、ピーク時には10台必要になった場合でも、オートスケーリング機能により自動的にワーカーを増減させることが可能です。

ステートフル接続に対してスケーラブルな設計を行う
ステートフル接続（例えばWebSocket）のスケールアウトは難しい問題です。各サーバーがコネクションを長時間保持しそのまま水平分割されるため、従来のステートレスなHTTPリクエストとは異なる設計上の制約が生じます。本章ではWebSocketを題材に、どのようにスケーラブルな設計を行うかを説明していきます。
なぜWebSocketのスケールアウトは難しいのか？
サーバー間で接続を共有しなければならない
WebSocketは、主にチャットのような複数のクライアント間でリアルタイム通信を行う用途で使われることが多いです。このようなケースでは、クライアントは複数存在するWebSocketサーバーのうちのいずれか1台に接続し、そのサーバーとの間に持続的なWebSocketコネクションを確立します。ただし、各サーバーは自分に接続されているクライアントの情報しか保持していません。そのため、あるサーバーに接続されたクライアントが送信したメッセージを、他のサーバーに接続されているクライアントにも届けるには、サーバー間でメッセージを共有する仕組みが必要になります。
大量のコネクション数がファイルディスクリプタやメモリの枯渇に直結
WebSocketはTCPを活用しています。そのため、各接続はOSリソース（メモリ、ファイルディスクリプタ）を専有し、接続数が増えるほどこれらのリソースを圧迫します。詳しくは下の2019年のGopherconのトークの記事やWebサーバにおけるソケット周りの知識を読むのがおすすめです。

TCPポートは枯渇する？
よくある誤解ですが、「TCPポートの範囲は0から65535なので、接続数の上限は65,536だ」というものですが実はこれは正しくありません。この数字は「使えるポート番号の範囲」を指しており、1つのサーバーポートが処理できる接続数の上限ではないわけです。
実際には、TCPのコネクションは「送信元IP・送信元ポート・宛先IP・宛先ポート」の4つの組み合わせによって一意に識別されます。そのため、OSの設定を適切に調整し、リソースを十分に確保すれば、1つのポートでも10万、場合によっては100万のWebSocketのコネクションを同時に処理することが可能です。

WebSocketのスケールアウト手法
ロードバランサー + Pub/Sub
複数のWebSocketサーバー間でのリアルタイムメッセージング課題に対して、ロードバランサーとPub/Subパターンを組み合わせることで、スケーラブルかつ効率的なメッセージ配信システムを実現できます。このアーキテクチャでは、Pub/Subブローカーを中央ハブとして活用し、サーバー間でのメッセージ共有を行います。
接続の振り分けと維持: ロードバランサーがクライアント接続を複数のWebSocketサーバー（WS1, WS2）に分散配置し、以降は各クライアントが割り当てられたサーバーとの間で継続的な双方向WebSocket通信を維持します。
メッセージの受信とPub/Sub配信: 特定のクライアント（例：Client1）からメッセージ（msg）が送信されると、接続先のWebSocketサーバー（WS1）がそのメッセージを受信し、即座にPub/Subブローカー（Redis、Apache Kafkaなど）の指定されたトピックにパブリッシュします。
全サーバーへの即時配信: Pub/Subブローカーは、サブスクライブしている全てのWebSocketサーバー（WS1, WS2を含む）に対してメッセージをFan-out方式で各サーバーへ即時配信します。
クライアントへの最終配信: 各WebSocketサーバーは、Pub/Subブローカーから受信したメッセージを、自身に接続されている全クライアント（Client2, Client3, Client4など）に対してWebSocket経由で配信し、全体的なリアルタイムメッセージング体験を完成させます。
双方向通信の継続: このプロセスは全てのクライアントからのメッセージに対して同様に実行され、どのサーバーに接続されているクライアントでも、システム全体の他の全てのクライアントとリアルタイムでメッセージ交換が可能になります。
例えば、LINE LIVE（2023年サービス終了）のリアルタイムコメント機能ではRedis Pub/Subを活用しています。チャットルームが複数サーバーに分割されており、各チャットルームが複数のサーバーにまたがって存在する場合でも、Redis ClusterのPub/Sub機能を利用してサーバー間でコメントを同期することで、同一チャットルーム内のすべての視聴者にコメントが届くようにしています。

ロードバランサー + Pub/Subの課題
Fan-outコスト
多くのWebSocketサーバーが、数多く同じPub/Subのトピックをサブスクライブする事でメッセージを高頻度で受け取りシステムへ高負荷を与える可能性があります。例えば、同じチャットルームに全員にメッセージPub/Sub経由で送る場合を考えます。Pub/Subでは購読している全てにFan-outされるので、最悪のケース全てのサーバーに分散している場合、全体にブロードキャストすることと同じ現象が起きます。
対策案としてはサーバーごとにチャットルームを一意に定めて、そこにユーザーのコネクションを集約させれば、Fan-outのスコープを狭めることができますが、これは単純なロードバランサーでは実現は難しいです。後述するコーディネーションサービスを用いると実現できます。
大量の同時接続に関するPub/Subのスケーラビリティ
単一のPub/Subブローカーでは、数十万から数億の同時接続においてスケーラビリティの問題が発生します。基本的にはシャーディングのアプローチを使ってトピックをPub/Subクラスタに分散させることでスケーラビリティを確保します。
例えば、チャットルームIDやユーザーIDのハッシュに基づいて、各ルームのメッセージが異なるPub/Subクラスタにルーティングされるように設計します。これにより、1つのブローカーに過度な負荷が集中するのを防ぎ、スループットを水平方向にスケールさせることが可能になります。
コーディネーションサービス + Consistent Hashing
より大規模で効率的な構成を実現したい場合にはZooKeeper、Consul、etcdといったコーディネーションサービス + Consistent Hashingを活用することで、より効率的な構成が可能になります。
コーディネーションサービスとはリーダー選出やサービスディスカバリなど分散システムで課題になっている問題を解決する専用のサービスです。実際中身はキーバリューストアなのですが、アプリケーションデータを管理するものではなく、分散システムのメタデータ（サーバーやサービスの情報など）を管理しています。今回のケースではコーディネーションサービスを各クライアントがどのWebSocketサーバーに接続すべきかを割り当てるサービスディスカバリーとして活用します。

コーディネーションサービスを使ったサービスディスカバリーの設計
ここではどのようにコーディネーションサービスを利用してサービスディスカバリを作成するか解説します。
WebSocketサーバーの自己登録: 各WebSocketサーバー（WS1, WS2, WS3）が起動時および定期的に、自身の状態情報（サーバーID、IPアドレス、ポート番号、現在の接続数、負荷情報、ヘルス状態）をCoordination Serviceに登録します。
Consistent Hash Ring Managerの構築: Coordination Serviceは登録されたWebSocketサーバー情報を基に、Consistent Hash Ring Managerは、各サーバー（ノード）のハッシュ位置を決定して、Coordination Serviceに登録します。これにより、クライアント-サーバーマッピングを実現でき、どのクライアントがどのWebSocketサーバーに接続するかを一意に判断することができます。
クライアント接続要求の処理: Service Discoveryが「どのWSに接続すべきか」のリクエストを受け取ると、Coordination ServiceからConsistent Hash Ringの状態を取得します。
サーバー選択とルーティング: 各クライアント（Client1, Client2, Client3）に対して、Consistent Hashingアルゴリズムに基づき最適なWebSocketサーバーを決定し、クライアントを適切なサーバー（WS1, WS2, WS3）に振り分けます。

次に、メッセージがどのように送信されるかを見ていきましょう。
同一サーバー内でのメッセージ配信: Client2およびClient3はClient1と同じWebSocketサーバー（WS1）に接続しているため、メッセージは直接転送されます。
異なるサーバーへのメッセージ転送: Client4はWS2に接続しているため、WS1がCoordination Serviceを参照してConsistent Hash Ringから計算されたClient4の接続先を確認し、WS2にメッセージを転送します。

コーディネーションサービス + Consistent Hashingの課題
スケーリング時のマッピング破綻
Consistent Hash Ringを使用したWebSocketサーバー構成において、サーバーのスケールイン・アウト時に発生する最大の課題は、既存のクライアント-サーバーマッピングの一部破綻です。
図に示されているように、新しくWS4ノードが追加された場合、クライアントC1の担当サーバーがWS1からWS4に変更されます。しかし、物理的にはC1はまだWS1に接続したままの状態です。この結果、他のクライアント（例：C4やC6）がC1にメッセージを送信する際、更新されたHash Ring参照によりWS4を宛先と判断するため、実際のC1（WS1に接続中）にメッセージが届かない問題が発生します。
この問題を解決するため、マイグレーション期間中はC1宛のメッセージを旧サーバー（WS1）と新サーバー（WS4）の両方に重複送信する必要があります。つまり、C1がWS1からWS4への接続移行を完了するまで、C1はどちらのサーバーにも存在する可能性があるため、両サーバーにリクエストを送信することで確実な配信を保証します。

WebSocketサーバーのスケールインとデプロイ戦略における再接続ストーム対策
なぜWebSocketはスケールインやデプロイメントは難しいのか？
WebSocketは長時間にわたって接続を維持するため、HTTPとは異なりサーバーを縮小（スケールイン）したり、新バージョンをデプロイしたりする際に、接続の移行を明示的に制御する必要があります。
このとき多くのクライアントが一斉に再接続を試みと、再接続ストーム（Reconnection Storm）が発生し、新しいサーバー群に過大な負荷をかけてしまいます。

グレースフルシャットダウンを実現するためには、以下の流れが基本になります。
クライアントに対して、アプリケーションレベルでclose frameを送信し、再接続を促します。
クライアントはclose frameを検知すると、即座に再接続処理を開始し、別のインスタンスに接続し直します。
この方式であれば、コネクションは明示的かつ安全に閉じられ、再接続ができます。
しかしコーディネーションサービス + Consistent Hashingを使っている場合には、スケールイン・アウトの再接続の際にサーバー（ノード）が増減しても移動させる接続数を最小限に抑えることができ、全体の影響を局所化できる利点がありますが、課題でも述べたようにはConsistent Hash Ringによる計算が一部ずれるので、慎重に検討しなければなりません。
WebSocketコネクションの段階的の移行
グレースフルシャットダウンを行うことで、クライアントは突然接続が切断されることなく、close frameを受け取った後に安全に再接続できるようになります。次に考慮すべきは、この再接続のトラフィックをいかに制御し、システム全体への負荷を最小限に抑えるかという点です。
例えば、古いサーバーAに存在するすべてのコネクションを一斉に切断すると、新しいサーバーBに再接続が集中し、瞬間的に高負荷となる可能性があります。そのため、コネクションは段階的に移行することが望ましいです。
具体的には、毎秒あたり数百接続ずつclose frameを送ることで、クライアントの再接続をバッチ単位に分散させることができます。これにより、各サーバーへの負荷を平準化しながら、安全に移行を進めることができます。
